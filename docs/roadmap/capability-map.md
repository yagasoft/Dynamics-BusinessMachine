# DBM Capability Map

This matrix is the source of truth for how sophistication capabilities mature across releases.

| Capability | Architectural role | `R1` foundation | `R2` designer/process productization | `R3` Dataverse-first operational completion | `R4` AI-assisted augmentation | `R5` Azure deferred extension | `R6` enterprise depth |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Reusable condition library | Shared process predicate layer | Define reusable condition model and references | Surface reusable conditions clearly in the visual designer and process preview | Use in live Dataverse-backed runtime operations | Suggest and refine conditions | Extend only if approved Azure services need shared policy evaluation | Govern reusable condition assets at scale |
| Explainability hooks | Why the system behaved that way | Add explainability hooks for branching, visibility, assignment, and status projection | Expose explainability in the designer and process-experience preview | Expose support-facing explanations in Dataverse-first live operations | AI summarizes and critiques logic paths | Add Azure explanations only for approved Azure-owned responsibilities | Enterprise-grade explainability across authoring and runtime |
| Form-state orchestration | Stateful UI behavior without form explosion | Define form-state foundations in model and designer | Author and preview form-state effects visually against the shared process shell | Use in portal and model-driven operational flows | AI suggests form-state improvements | N/A unless an approved Azure host needs form-state context | Broader optimization of form-state patterns |
| Multi-table process view | Process modeling across related records | Establish multi-table modeling foundation | Keep mapping-first boundaries visible in the designer without taking full form ownership | Use in live end-to-end reference solution | AI identifies missing cross-table coverage | N/A unless approved Azure integration needs multi-record projection | Richer multi-table modeling depth |
| Synthesis preview/apply/diff baseline | Safe generated Dataverse authoring | Add preview, apply, and diff baseline for forms and columns | Extend to process-host assets, placements, and emitted renderer bindings | Use in live environments with operator workflows | AI explains and proposes synthesis changes | Add Azure deployment preview only for approved Azure assets | Enterprise governance, approval, and drift control |
| Drift awareness | Detect divergence between intent and reality | Establish initial drift-awareness baseline | Keep designer, emitted assets, and process-host patches aligned | Use in support and admin operations | AI highlights risky drift patterns | Include Azure drift only for approved Azure assets | Managed drift control at scale |
| Runtime event foundation | Structured runtime events for later ops | Emit runtime event foundation | Feed the shared process-experience snapshot and cross-form messaging | Power timeline, audit, and support surfaces from Dataverse-owned events | AI reviews event patterns and anomalies | Bridge Azure-owned events back to Dataverse records when Azure is approved | Portfolio-scale diagnostics and optimization |
| Work management layer | Operational control of human work | Design for ownership and task semantics | Keep the process shell ready to display future work-management actions | Deliver Dataverse-backed inboxes, queues, reassignment, delegation, escalation, and SLA timers | AI helps prioritize and analyze work patterns | Add Azure workers only where Dataverse jobs cannot reasonably own the work | Deeper control-plane and optimization depth |
| Timeline and audit trail | Supportable history of runtime behavior | Define event shapes and references | Keep snapshot and renderer seams ready for future timeline projection | Deliver first-class Dataverse-owned timeline and audit surfaces | AI summarizes histories and highlights anomalies | Include Azure-owned event telemetry only for approved Azure services | Enterprise-grade history analysis |
| Runtime observability | Performance and failure insight | Define observability-friendly runtime outputs | Add component, visual, and host-placement validation for designer/process UX | Deliver Dataverse-first observability baseline and support diagnostics | AI suggests performance and cost improvements | Add Azure telemetry only when Dataverse/Power Platform surfaces cannot meet the need | Advanced analytics and optimization |
| Process simulator | Safe pre-deployment and support analysis | Record required simulation hooks only | Keep graph and snapshot structures simulation-friendly | Use operational data to support later simulation | AI helps prepare scenarios | N/A unless simulation needs approved Azure compute | Deliver deterministic simulation and replay debugger |
| Replay debugger | Diagnose real runtime behavior | Record replay-friendly event foundation only | Preserve renderer seams and snapshot structure for later replay | Preserve enough audit data for replay | AI summarizes replay outcomes | N/A unless replay needs approved Azure compute/storage | Deliver support-facing replay debugger |
| Reusable process building blocks | Reduce duplicated modeling | Record reuse-oriented process boundaries | Make graph-first authoring ready for later abstraction and reuse | Prove repeatable scenario patterns | AI suggests reusable abstractions | N/A unless approved Azure service contracts need reusable process fragments | Deliver subflows, templates, and policy packs |
| Requirement-to-process drafting | AI-assisted authoring | N/A | N/A | N/A | Generate editable draft processes, metadata, forms, and process-experience proposals | N/A unless AI integration has an approved Azure dependency | Deepen with enterprise reusable patterns |
